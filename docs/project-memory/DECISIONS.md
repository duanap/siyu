# 架构与产品决策

## ADR-001：模块化单体

状态：已确认
决定：NestJS 单体按业务模块拆分，Worker 独立进程，共用领域服务。
原因：当前规模不需要微服务，优先保证一致性、测试和部署简单。

## ADR-002：金额使用整数分

状态：已确认
决定：数据库和内部 API 使用整数分。
原因：避免浮点计算误差。
禁止：FLOAT、DOUBLE 和 JavaScript 浮点累计财务金额。

## ADR-003：账本与私有数据双重隔离

状态：已确认
决定：普通账目以账本隔离；工资和个人借贷以用户隔离。
原因：情侣共同记账不等于共享全部个人财务。

## ADR-004：特殊业务通过来源键关联普通账目

状态：已确认
决定：工资、借贷处理和周期实例生成账目时使用 `source_type + source_id` 唯一关联。
原因：防止重复入账并支持追踪来源。

## ADR-005：周期任务使用执行实例

状态：已确认
决定：周期规则与每期实例分表，`rule_id + scheduled_date` 唯一。
原因：支持重试、确认、跳过、错误记录和幂等。

## ADR-006：Ant Design Vue 是基础组件而非完整视觉

状态：已确认
决定：表单和通用交互使用 Ant Design Vue，核心手机页面和卡片二次封装或自定义。
原因：避免手机端变成缩小的管理后台。

## ADR-007：工资余额按发薪周期

状态：暂定已确认
决定：默认从工资到账日至下一预计发薪日前一天。
原因：比自然月更符合“这次工资还剩多少”的用户认知。

## ADR-008：管理后台不直接改财务明细

状态：已确认
决定：后台以状态、关系、任务和审计查询为主。
原因：降低隐私、误操作和内部越权风险。

## ADR-009：SIYU 品牌与工程命名

状态：已确认
决定：品牌唯一事实来源为 `docs/product/BRAND_IDENTITY.md`；根包名 `siyu`，服务和包使用
`siyu` / `@siyu` 前缀，既有数据库业务表名和 `/api/v1` 路径不因品牌迁移而改变。
原因：统一产品展示和工程标识，同时避免无业务收益的兼容性破坏。

## ADR-010：Prisma 特殊约束由受审迁移 SQL 固化

状态：已确认
决定：Prisma Schema 作为模型基线；部分唯一索引、CHECK 和复合归属外键在 create-only 初始迁移中
以 PostgreSQL SQL 固化，并由一次性测试数据库自动验证。
原因：这些约束无法全部由 Prisma Schema 元数据直接表达，但不能降级为仅应用层校验。

## ADR-011：客户端幂等键作用域

状态：已确认
决定：客户端幂等键按“用户 + 操作 + Key”隔离；相同载荷重放原结果，不同载荷返回 409。
原因：允许不同用户或不同操作安全复用客户端生成值，同时阻止单个操作重复写入。

## ADR-012：双认证与可撤销会话

状态：已确认
决定：保留 QQ OAuth 并增加邮箱密码认证；邮箱与 QQ 身份不自动合并。Access Token 短时有效，
Refresh Token 以数据库摘要记录、每次轮换并在重放时撤销会话。认证首次建号继续原子初始化个人账本。
原因：满足基础账号能力，同时避免自动身份合并导致账号接管，并为禁用、退出和改密提供服务端撤销边界。

## ADR-013：最小关联表 RBAC

状态：已确认
决定：角色、权限、用户角色和角色权限使用关联表；默认注册仅授予 USER，ADMIN 仅由受控运维流程授予。
原因：管理端权限必须来自数据库和服务端 Guard，且需支持后续扩展而不暴露客户端提权入口。

## ADR-014：情侣账本退出、转移与解散

状态：已确认
决定：普通成员可以退出；所有者不能直接退出，必须先把所有权原子转移给另一名有效成员，或由所有者解散账本。解散只撤销访问并结束成员和邀请状态，历史财务数据保留用于追溯。
原因：情侣账本必须始终有明确责任人，且关系变化不能破坏历史财务记录或产生短暂无所有者、双所有者状态。

## ADR-015：情侣邀请单活与一次性凭证

状态：已确认
决定：每个情侣账本同一时间只保留一个有效待处理邀请；新建邀请撤销旧邀请，接受后撤销其他待处理邀请。数据库只保存凭证摘要，明文凭证仅在创建响应中返回一次；创建请求使用用户、操作和幂等键隔离。
原因：减少多邀请码并发加入的歧义，避免凭证泄露，并满足重复点击和网络重试时不重复创建邀请。

## ADR-016：自定义分类按账本作用域隔离

状态：已确认
决定：系统分类全局可用；个人自定义分类仅用于个人账本；情侣自定义分类归情侣账本，双方共享使用，创建者可管理自己的分类，账本所有者可管理全部情侣分类。历史账目保持分类展示语义，成员退出后不再授予分类权限。
原因：避免将个人分类和使用习惯无意共享给情侣成员，同时保证共同账目分类口径一致。

## ADR-017：系统分类采用全局模板、账本内实例

状态：已确认
决定：默认系统分类以版本化 `templateKey` 维护全局模板目录，但实际 Category 记录必须按账本实例化；
初始化以 `(ledgerId, templateKey)` 保证幂等。系统分类内容不可由用户修改，OWNER 可排序和启停；
情侣 MEMBER 继续按 ADR-016 管理自己创建的自定义分类，不具备全局排序和系统分类操作权限。
原因：账本实例能直接建立分类、账本和类型的数据库边界，允许各账本独立排序/启停，又不牺牲默认目录升级和初始化幂等。

## ADR-018：Entry 使用不可变创建哈希、乐观锁和来源维护边界

状态：已确认
决定：手工 Entry 以版本化规范载荷的 SHA-256 `createRequestHash` 判断创建重放，该字段创建后不可变；
PATCH 和 DELETE 使用整数 `version/expectedVersion` 乐观锁。正常查询使用 `deleted_at IS NULL` 部分索引；
普通 API 不修改或删除非 MANUAL 来源账目。软删除重试仅在完整重验账本可见性、当前操作者权限、来源类型，
且墓碑版本恰为请求版本加一时幂等成功。能力字段只按当前操作者计算。
原因：后续可编辑字段不能反向改变原创建请求事实；乐观锁避免财务数据静默覆盖；来源边界和权限优先的
墓碑重试同时保证跨模块一致性、防枚举和网络重试安全。
