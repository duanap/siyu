# 验收标准

## 用户与账本

- **AC-USER-001**：首次邮箱注册或 QQ 登录成功后，用户、个人账本和所有者成员同时存在；任一创建失败时全部不落库。
- **AC-LEDGER-001**：重复登录不会创建第二个个人账本。
- **AC-AUTH-001**：登录失败不泄露邮箱是否存在；密码和令牌不出现在响应与日志中。
- **AC-AUTH-002**：Refresh Token 轮换后旧令牌失效；旧令牌重放撤销所属会话，并发刷新不产生两个有效后继。
- **AC-AUTH-003**：修改密码撤销其他会话并轮换当前会话；重置密码撤销全部会话。
- **AC-AUTH-004**：密码重置请求对存在和不存在邮箱返回一致结果，令牌过期、无效或重复使用均被拒绝。
- **AC-AUTH-005**：普通用户不能访问管理端校验接口，管理员可以访问。
- **AC-COUPLE-001**：用户可创建情侣账本并生成有时效的邀请码。
- **AC-COUPLE-002**：邀请被第二位用户接受后，第三位用户无法加入。
- **AC-COUPLE-003**：已加入其他情侣账本的用户无法接受新邀请。
- **AC-COUPLE-004**：邀请创建者无法接受自己的邀请。
- **AC-COUPLE-005**：双方可查看共同账目，不能通过接口获取对方工资或个人借贷。
- **AC-COUPLE-006**：普通成员不能修改对方账目；所有者可管理全部共同账目。
- **AC-COUPLE-007**：退出或解散后，原成员再次访问详情接口返回无权限或资源不存在。
- **AC-COUPLE-008**：普通成员退出后账本仍由所有者持有；所有者直接退出被拒绝，转移所有权后原所有者可以退出。
- **AC-COUPLE-009**：所有权转移并发或失败时，账本所有者和双方成员角色不会出现部分更新或双所有者。
- **AC-COUPLE-010**：解散后所有成员立即失去访问权限、待处理邀请失效，历史账本与财务记录未被物理删除。
- **AC-COUPLE-011**：重复创建同一邀请请求返回同一结果；新邀请会使旧待处理邀请失效，明文凭证只在创建响应中返回。
- **AC-COUPLE-012**：个人自定义分类不能用于情侣账本；情侣自定义分类双方可用，退出后不可继续访问或管理。

## 普通记账

### 分类

- **AC-CATEGORY-001**：邮箱注册、首次 QQ 建号和情侣账本创建后各自获得 9 个支出与 7 个收入默认分类；重放或并发初始化不重复。
- **AC-CATEGORY-002**：个人和情侣账本分类记录彼此隔离；跨账本同名允许，同账本同类型启用分类重名被拒绝。
- **AC-CATEGORY-003**：OWNER 可管理全部分类；情侣 MEMBER 只能管理自己创建的自定义分类，不能排序或操作系统/他人分类。
- **AC-CATEGORY-004**：创建幂等重放返回同一分类，不同载荷复用幂等键返回冲突；快速重复提交不会创建重复分类。
- **AC-CATEGORY-005**：排序始终按 `sortOrder,id` 稳定返回，完整顺序重写为 100 间隔，相同排序重复提交无额外审计。
- **AC-CATEGORY-006**：分类停用后历史账目和周期规则仍可读取；新 Entry/RecurringRule 绑定停用、跨账本或错误类型分类时数据库拒绝。
- **AC-CATEGORY-007**：非法名称、图标、颜色、排序、系统分类修改和跨账本访问返回统一且不泄露资源的错误。
- **AC-CATEGORY-008**：分类页面在 320px、375px、480px 的日间/暗色下覆盖加载、空、错误、长文本、权限、重复提交和启停确认，无横向滚动。

- **AC-ENTRY-001**：用户可新增收入和支出，并在明细和统计中正确出现。
- **AC-ENTRY-002**：金额精确到分，连续累计不出现浮点误差。
- **AC-ENTRY-003**：分类类型与账目类型不匹配时保存失败。
- **AC-ENTRY-004**：用户不能通过修改 ID 查看或编辑无权访问的账目。
- **AC-ENTRY-005**：删除后账目不出现在明细、搜索、统计和导出中。
- **AC-ENTRY-006**：重复点击保存只产生一笔账目。
- **AC-ENTRY-007**：320px 宽度下新增、详情和列表页面无横向滚动。
- **AC-ENTRY-008**：个人 OWNER、情侣 OWNER 和 MEMBER 按 BR-ENTRY-009 查看和管理账目；越权 ID、退出关系、
  非成员和解散账本均不泄露记录是否存在。
- **AC-ENTRY-009**：相同幂等键和原始创建载荷在账目后续被修改后仍重放同一记录；不同载荷、legacy 记录或
  已删除原记录返回幂等冲突，并发提交只产生一笔账目。
- **AC-ENTRY-010**：PATCH 和 DELETE 的版本冲突返回 `ENTRY_VERSION_CONFLICT`；无实际变化的 PATCH 不递增版本、
  不新增审计。
- **AC-ENTRY-011**：软删除后列表、详情和搜索不可见；合法网络重试幂等成功但仍经过权限、来源和版本校验，
  且只产生一条删除审计。
- **AC-ENTRY-012**：非 MANUAL 账目可按权限查看但普通修改和删除返回 `ENTRY_SOURCE_MANAGED`。
- **AC-ENTRY-013**：金额、日期、支付方式、备注、分类归属/类型/启用状态和幂等键在 DTO 与数据库层均受约束。
- **AC-ENTRY-014**：列表支持月份、类型、分类、创建人、备注关键词和分页，按业务日期、创建时间、ID 稳定倒序，
  停用分类仍能在历史账目中展示且不产生 N+1 查询。
- **AC-ENTRY-015**：客户端以字符串和整数算法完成元到分转换，拒绝零、负数、指数、三位小数和超安全整数；
  `amountCent` 始终为正数，展示符号由账目类型决定。
- **AC-ENTRY-016**：快速记账失败重试保持同一幂等键，成功后才轮换；重复点击只发出一次有效创建请求，
  网络失败保留全部表单字段。
- **AC-ENTRY-017**：明细账本、月份、筛选和分页深度可由 URL 恢复；深度有固定上限，取消、无下一页和非法页码
  均安全终止，旧请求不得覆盖新条件。
- **AC-ENTRY-018**：详情的编辑和删除只消费服务端能力；版本冲突不强制覆盖，非 MANUAL 来源账目明确只读，
  `from` 仅允许 Router 解析后路由名为 `entries` 的站内目标。
- **AC-ENTRY-019**：新增、列表和详情在 320px、375px、480px 的日间与暗色模式下无横向滚动，
  支持键盘焦点、44px 点击区、长文本、加载、空、错误、冲突和提交状态。

## 借贷

- **AC-DEBT-001**：用户可分别创建“我欠别人的”和“别人欠我的”记录。
- **AC-DEBT-002**：借贷本金不自动进入普通收入或支出统计。
- **AC-DEBT-003**：支持多次部分还款/收款，剩余金额计算正确。
- **AC-DEBT-004**：处理金额超过剩余金额时接口拒绝。
- **AC-DEBT-005**：最后一次处理后剩余为零并自动标记已结清。
- **AC-DEBT-006**：到期未结清的记录显示逾期天数。
- **AC-DEBT-007**：选择同步账目后只生成一笔正确类型的普通账目。
- **AC-DEBT-008**：重复提交同一处理请求不重复扣减或生成账目。
- **AC-DEBT-009**：存在已同步账目的借贷删除被拒绝且账目保留；没有同步账目的借贷及未同步处理记录可软删除。
- **AC-DEBT-010**：借贷列表基于完整本人数据正确展示总负债、总待收款、本月已还、本月已收、未来 7 日
  即将到期和已逾期数量；完整性不足时明确显示统计不可用。
- **AC-DEBT-011**：借贷新增、列表和详情在 320px、375px、480px 的日间与暗色模式下无横向滚动，
  覆盖加载、空、错误、404、长文本、逾期、不可删除、超额处理、快速重复提交和浏览器返回/刷新恢复。

## 周期记账

- **AC-RECUR-001**：可创建每月、每年和固定期数规则。
- **AC-RECUR-002**：自动规则到期后产生一条执行实例和一笔账目。
- **AC-RECUR-003**：确认规则到期后只产生待确认实例，确认后才入账。
- **AC-RECUR-004**：Worker 重跑、重试和并发执行均不重复入账。
- **AC-RECUR-005**：暂停期间不生成新实例。
- **AC-RECUR-006**：达到总期数后状态为已完成，下一执行日为空。
- **AC-RECUR-007**：生成失败可在后台看到错误和重试次数。
- **AC-RECUR-008**：情侣共同规则对双方可见，个人规则仅本人可见。
- **AC-RECUR-009**：暂停后恢复从当天或之后的下一期继续，不生成暂停期间的遗漏实例。
- **AC-RECUR-010**：1 月 29 至 31 日月度规则、2 月 29 日年度规则和跨年间隔均以开始日期为锚，短月取月末且后续日期不漂移。
- **AC-RECUR-011**：待确认实例跳过后不生成账目、不重复推进期数且不能再确认；重复跳过返回同一终态。
- **AC-RECUR-012**：确认可使用与规则默认值不同的正整数分金额，且只更新当前实例；相同幂等请求重放同一账目，
  不同金额复用幂等键或并发确认不会产生第二笔账目。
- **AC-RECUR-013**：个人规则仅本人可见；情侣双方可见，创建人和账本 OWNER 按权限管理，普通成员不能修改
  对方创建的规则，退出、解散和可猜测 ID 不泄露资源。
- **AC-RECUR-014**：规则创建重复提交只产生一条规则；更新不改写既有实例，总期数小于已物化期数、停用/
  跨账本/类型错误分类、已有实例后的计划核心字段变化和失效创建人均被拒绝。
- **AC-RECUR-015**：按用户时区跨 UTC 日期边界扫描不漏单；重复扫描、多 Worker、任务重跑和延迟旧任务不重复
  实例、账目或通知。临时失败按指数退避重试，业务状态失败只记录一次并进入最终失败；确认实例只通知当前
  规则创建人和情侣账本所有者，日志不包含金额、规则名称或底层数据库错误正文。

## 工资

- **AC-SALARY-001**：用户可创建工资档案并复制上月模板。
- **AC-SALARY-002**：收入项目、扣除项目、应发和实发计算一致。
- **AC-SALARY-003**：同一档案和月份重复创建时返回冲突。
- **AC-SALARY-004**：未到账记录不生成普通收入。
- **AC-SALARY-005**：确认到账后生成金额等于实发工资的收入。
- **AC-SALARY-006**：重复确认到账不产生第二笔收入。
- **AC-SALARY-007**：年度养老、医疗、失业、公积金和个税累计与月度明细之和一致。
- **AC-SALARY-008**：工资余额只统计选定工资周期内符合条件的个人支出。
- **AC-SALARY-009**：剩余天数为零时不出现除零错误。
- **AC-SALARY-010**：情侣成员无法通过任何情侣接口或可猜测 ID 获取对方工资。
- **AC-SALARY-011**：发薪日 29 至 31 日在短月份按当月最后一天预测，覆盖平年和闰年二月。
- **AC-SALARY-012**：同一用户并发创建工资档案时最多一条成功；非本人、失效用户和可猜测档案 ID 不泄露工资事实。
- **AC-SALARY-013**：档案默认模板可创建和原子替换，允许零金额模板项；模板变化不改写既有月度记录。
- **AC-SALARY-014**：月度记录可使用显式正金额项目，或复制紧邻上月的项目；缺少紧邻上月、同时提交两种来源、
  项目代码重复或扣除大于应发时均被拒绝。
- **AC-SALARY-015**：档案和记录创建相同幂等键同载荷重放原结果，不同载荷返回冲突；不同键重复月份返回
  `SALARY_MONTH_DUPLICATE`，并发请求只产生一条有效记录。
- **AC-SALARY-016**：月度项目创建和替换后，应发、扣除和实发与项目精确一致；事务失败不留下部分项目或旧汇总。
- **AC-SALARY-017**：已到账记录不能通过月度更新接口修改；列表、详情和更新均严格限定当前用户，情侣成员
  和另一工资用户无法通过 ID、档案或月份读取任何工资或项目明细。
- **AC-SALARY-018**：确认到账提交 `paidDate`、`syncEntry` 和到账幂等键；不生成收入时记录保持 `entryId=null`，
  生成收入时只在本人有效个人账本创建一条 `income.salary` 收入，其金额、日期、创建人和 `SALARY` 来源与工资
  记录一致。个人账本或工资分类不可用、审计或任一步写入失败时，状态、明细和幂等事实均不落库。
- **AC-SALARY-019**：同键同载荷重复或并发确认返回同一到账结果且最多一条来源收入；同键不同载荷返回
  `IDEMPOTENCY_CONFLICT`，不同键再次确认返回 `SALARY_ALREADY_PAID`。失效用户、情侣成员和其他用户通过
  可猜测 ID 均不能确认或发现工资事实；数据库直接修改已到账事实或 `SALARY` 来源财务字段被约束拒绝。
- **AC-SALARY-020**：年度汇总的应发、扣除、实发和六类专项累计与该年工资记录/项目精确一致；趋势始终返回
  1 至 12 月且缺失月份为 0，跨年记录不串入，月均实发使用有效记录月份数并保持整数分。
- **AC-SALARY-021**：工资余额按用户时区选择不晚于今天的最近到账记录；短月、闰年二月和跨年时下一预计
  发薪日正确。周期内本人 PERSONAL 账本的固定/日常支出拆分、已支出和剩余金额精确一致，情侣支出被排除。
- **AC-SALARY-022**：没有已到账工资时返回可识别空态；周期过期时剩余天数为 0 且日均可用金额为 `null`，
  负余额安全返回。另一用户、情侣成员和失效可猜测资源不得影响或泄露年度汇总与工资余额。
- **AC-SALARY-023**：工资首页只展示本人档案、服务端工资余额、当前月记录和服务端年度汇总；没有档案时提供
  首次设置，不展示伪造金额或空图表。档案模板允许 0 分，名称、发薪日、项目代码、重复项目和至少一项收入
  在提交前校验；创建失败重试保持同一幂等键，成功后才轮换。
- **AC-SALARY-024**：月度页按完整年度分页匹配目标月份；新建可使用模板中的正金额项目或由服务端复制紧邻
  上月，0 分模板项不写入月度记录。创建和到账快速重复点击最多发出一次有效写请求，失败重试保持各自幂等键；
  到账前二次确认日期和是否同步收入，到账后严格只读并提示不可修改事实。
- **AC-SALARY-025**：年度页直接消费服务端 12 个月趋势与专项累计，仅存在记录时渲染图表，始终提示“数据来自
  工资记录，不代表官方账户余额”。工资三页在 320px、375px、480px 的日间和暗色模式下无横向滚动，覆盖
  加载、无档案、空年度、网络错误、无权限、长文本、浏览器返回和刷新恢复，所有可见交互区不小于 44px。

## 攒钱目标

- **AC-SAVING-001**：用户可创建个人目标和有效情侣账本的共同目标。
- **AC-SAVING-002**：新增、修改、删除存入记录后汇总金额正确。
- **AC-SAVING-003**：达到目标金额后自动标记已完成。
- **AC-SAVING-004**：情侣目标分别显示双方贡献。
- **AC-SAVING-005**：无权限用户无法查看或写入目标。
- **AC-SAVING-006**：并发存入不会丢失金额或重复累计。
- **AC-SAVING-007**：初始金额计入创建者贡献；金额从已完成回落到目标以下后状态恢复进行中。
- **AC-SAVING-008**：目标创建和存入创建同键同载荷返回同一资源且不重复累计，同键不同载荷返回 409；软删除
  后重放旧键不复活资源。并发不同键存入全部保留，`savedCent` 与初始金额加有效存入逐笔合计精确一致。
- **AC-SAVING-009**：个人目标仅本人可见；情侣双方有效成员可读和添加本人存入，只有 OWNER 可更新或删除
  目标，每位成员只能更新/删除本人存入。退出、解散、跨账本和可猜测 ID 均不泄露目标或贡献事实。
- **AC-SAVING-010**：创建时初始金额达到目标直接完成；修改目标金额、修改或删除存入可使状态在 `ACTIVE` 与
  `COMPLETED` 间双向切换。删除目标后列表/详情不可见、贡献历史保留且所有后续写入被拒绝。
- **AC-SAVING-011**：列表和详情返回精确整数分的目标金额、初始金额、已存、剩余、整数万分比进度、成员贡献
  和服务端能力；详情的有效存入按业务日期和创建时间稳定倒序，初始金额只出现在创建者贡献中而不是伪造记录。
- **AC-SAVING-012**：移动端目标列表支持个人与有效情侣账本切换和 URL/本地选择恢复，完整读取分页并直接展示
  服务端目标金额、已存、剩余、整数万分比进度、状态、日期与贡献口径；空数据、加载、网络失败和长名称均有
  明确状态，不使用客户端浮点或增量重算目标事实。
- **AC-SAVING-013**：移动端目标创建与存入创建分别在失败重试前保持稳定幂等键，并在提交中禁止快速重复请求；
  金额使用精确元分转换。目标修改/删除只显示于 `canManage`，添加存入只显示于 `canContribute`，存入编辑/删除
  只显示于该记录的 `canEdit/canDelete`，删除动作必须二次确认，操作成功后重新读取服务端详情。
