# 业务规则

规则是实现和验收的最高业务事实来源。修改规则必须记录 ADR 和变更原因。

## 用户与账本

- **BR-USER-001**：首次邮箱注册或 QQ 登录必须在同一事务中创建用户、唯一个人账本及其所有者成员。
- **BR-USER-002**：被禁用用户不得创建或修改业务数据。
- **BR-USER-003**：邮箱与 QQ 身份不得仅凭相似资料自动合并。
- **BR-AUTH-001**：Refresh Token 每次使用必须原子轮换，旧令牌重放时撤销所属会话。
- **BR-AUTH-002**：密码重置令牌只保存不可逆摘要、短时有效且只能成功使用一次。
- **BR-AUTH-003**：注册用户默认仅有 USER 角色，客户端参数不得授予 ADMIN。
- **BR-LEDGER-001**：每个用户有且只有一个有效个人账本。
- **BR-LEDGER-002**：个人账本只能有一个有效成员，成员必须是所有者本人。
- **BR-COUPLE-001**：情侣账本最多两名有效成员。
- **BR-COUPLE-002**：一个用户同一时间最多加入一个有效情侣账本。
- **BR-COUPLE-003**：创建者是情侣账本所有者，另一方是成员。
- **BR-COUPLE-004**：邀请必须有有效期、状态和唯一随机凭证。
- **BR-COUPLE-005**：邀请创建者不能接受自己的邀请。
- **BR-COUPLE-006**：第二名成员加入和邀请状态更新必须在同一事务中完成。
- **BR-COUPLE-007**：解散后双方立即失去访问权限。
- **BR-COUPLE-008**：普通成员只能修改或删除自己创建的共同账目；所有者可管理全部共同账目。
- **BR-COUPLE-009**：工资和个人借贷不得通过情侣账本接口返回。
- **BR-COUPLE-010**：普通成员可主动退出；退出必须将成员状态改为已退出并立即撤销访问权限，历史成员和财务记录保留用于追溯。
- **BR-COUPLE-011**：所有者不得直接退出；存在另一名有效成员时必须先原子转移所有权，只有所有者可以解散账本。
- **BR-COUPLE-012**：所有权转移只能指向同一情侣账本的另一名有效成员，并在同一事务中交换双方角色、更新账本所有者并写审计日志。
- **BR-COUPLE-013**：解散必须在同一事务中将账本标记为已解散、结束全部有效成员关系、撤销待处理邀请并写审计日志；历史财务记录保留但不再对原成员开放。
- **BR-COUPLE-014**：同一情侣账本同一时间只保留一个有效待处理邀请；创建新邀请时撤销旧邀请，接受成功后撤销其余待处理邀请。
- **BR-COUPLE-015**：系统分类对个人和情侣账本通用；个人自定义分类不得用于情侣账本，情侣账本内创建的自定义分类归该账本共享。
- **BR-COUPLE-016**：情侣自定义分类双方均可查看和使用；创建者可修改或停用自己创建的分类，账本所有者可管理全部情侣自定义分类。成员退出或账本解散后，历史账目保留分类快照语义，但原成员不得继续使用或管理该分类。

## 金额、日期与删除

- **BR-MONEY-001**：数据库和内部 API 金额使用整数分。
- **BR-MONEY-002**：金额必须大于零，除非明确业务规则允许冲正。
- **BR-MONEY-003**：展示层负责分与元转换，禁止用浮点数累计财务金额。
- **BR-DATE-001**：业务发生日使用 `DATE`，系统时间使用 UTC 时间戳。
- **BR-DATE-002**：统计日期按用户时区解释。
- **BR-DELETE-001**：财务业务数据默认软删除。
- **BR-DELETE-002**：软删除记录不得进入正常查询和统计。
- **BR-DELETE-003**：删除来源业务记录必须处理其关联账目并写审计日志。

## 普通账目与分类

- **BR-ENTRY-001**：账目必须归属于用户有权访问的有效账本。
- **BR-ENTRY-002**：账目类型只能是收入或支出。
- **BR-ENTRY-003**：账目创建人必须是目标账本有效成员。
- **BR-ENTRY-004**：自动生成账目使用 `source_type + source_id` 唯一约束。
- **BR-ENTRY-005**：普通手工账目 `source_type` 为 `MANUAL`。
- **BR-ENTRY-006**：删除账目后相关统计缓存必须失效。
- **BR-ENTRY-007**：手工账目创建按“当前用户 + 创建操作 + 幂等键”隔离；不可变 `createRequestHash`
  必须基于规范化原始创建载荷计算，后续修改不得改变其幂等判断事实。
- **BR-ENTRY-008**：账目修改和删除使用整数版本乐观锁；版本不一致必须冲突，不允许静默最后写入覆盖。
- **BR-ENTRY-009**：个人账本所有者和情侣账本所有者可管理账本内全部手工账目；情侣普通成员只能修改、
  删除自己创建的手工账目。能力字段只按当前操作者计算，创建人被禁用不取消有效 OWNER 的管理权限。
- **BR-ENTRY-010**：普通账目 API 不得修改或删除非 `MANUAL` 来源账目；工资、借贷和周期来源由各自领域维护。
- **BR-ENTRY-011**：普通删除只执行软删除；相同版本的网络重试必须重新校验账本可见性、操作者权限、
  来源类型和墓碑版本，且不得重复写删除审计。
- **BR-ENTRY-012**：支付方式只允许 `CASH`、`WECHAT`、`ALIPAY`、`BANK_CARD`、`OTHER` 或空值。
- **BR-ENTRY-013**：备注必须裁剪且最长 500 个字符；空备注规范化为无备注。
- **BR-ENTRY-014**：创建、实际修改和首次删除账目必须与业务写入在同一事务内审计；审计不得保存完整备注、
  幂等载荷或认证凭证。
- **BR-ENTRY-015**：非成员、已退出成员和已解散账本不得获知账目是否存在；普通查询必须排除软删除记录。
- **BR-CATEGORY-001**：分类类型必须与账目类型一致。
- **BR-CATEGORY-002**：已被使用的分类只能停用，不能物理删除。
- **BR-CATEGORY-003**：系统分类不可被普通用户改名或删除。
- **BR-CATEGORY-004**：系统分类是全局模板目录；每个有效账本保存独立分类实例，个人和情侣账本不得共享分类记录。
- **BR-CATEGORY-005**：首次邮箱注册、首次 QQ 建号和创建情侣账本必须在原事务中幂等初始化 16 个版本 1 默认分类。
- **BR-CATEGORY-006**：分类名称必须裁剪且为 1–50 个字符；同账本同收支类型内启用分类名称忽略大小写唯一。
- **BR-CATEGORY-007**：系统分类内容不可修改；账本所有者可排序、停用和重新启用系统分类及全部自定义分类。
- **BR-CATEGORY-008**：情侣普通成员可创建、修改、停用和重新启用自己创建的自定义分类，不可操作系统分类、他人分类或全局排序。
- **BR-CATEGORY-009**：分类创建按创建者和幂等键隔离；重复载荷返回原结果，不同载荷复用幂等键必须冲突。
- **BR-CATEGORY-010**：分类排序按账本和收支类型串行化，完整顺序规范化为 100、200……；相同顺序重复提交无副作用。
- **BR-CATEGORY-011**：停用分类不破坏历史账目和周期规则引用，但新建或更换引用时必须拒绝停用、跨账本或类型不匹配的分类。
- **BR-CATEGORY-012**：创建、修改、实际排序变化、启用和停用分类必须写审计日志；非成员和失效成员按资源不可见处理。

## 统计

- **BR-STAT-001**：首页与统计只聚合当前用户有权访问的有效账本内未软删除账目，按账目业务日期归属月份；
  月份缺省为用户时区当前月。
- **BR-STAT-002**：统计聚合全过程使用整数分；结余等于收入减支出，分类占比按支出金额计算并以万分比整数返回。
- **BR-STAT-003**：当前月日均支出按用户时区当月已过自然日计算，过去月份按整月天数计算，未来月份为零；
  金额除法向下取整到分。
- **BR-STAT-004**：趋势必须返回所选自然月的完整逐日序列；情侣成员对比按账目创建人归属，个人账本只返回本人。

## 借贷

- **BR-DEBT-001**：借到的钱只增加负债，不自动算作普通收入。
- **BR-DEBT-002**：借给别人的钱只增加待收款，不自动算作日常消费支出。
- **BR-DEBT-003**：单次还款或收款金额不得超过剩余金额。
- **BR-DEBT-004**：新增处理记录、更新汇总和可选生成账目必须在同一事务中完成。
- **BR-DEBT-005**：剩余金额为零时自动标记已结清。
- **BR-DEBT-006**：到期日早于当前业务日期且未结清时标记已逾期。
- **BR-DEBT-007**：同一还款记录最多关联一笔普通账目。
- **BR-DEBT-008**：个人借贷仅本人可访问。
- **BR-DEBT-009**：借贷存在任何已同步普通账目的处理记录时禁止删除；仅不存在同步账目的借贷可连同未同步处理记录软删除，关联普通账目不得由借贷删除操作解绑或删除。
- **BR-DEBT-010**：借贷创建和处理请求按用户、操作和幂等键隔离；相同规范载荷重放原结果，不同载荷或已删除原结果返回幂等冲突。
- **BR-DEBT-011**：借贷汇总基于本人全部未删除借贷；当前总负债和总待收款按未结清剩余金额汇总，
  本月已还和本月已收按处理记录业务日期归月。即将到期指按用户时区业务日期从今天起 7 个自然日内
  （含今天）到期且未结清的记录，已逾期按服务端 `overdueDays > 0` 判断。
- **BR-DEBT-012**：客户端只有在完整读取借贷分页及用于月度汇总的处理记录后才展示汇总；任一分页或详情
  请求失败时必须标记统计不可用，不得用已加载子集冒充完整统计。

## 周期记账

- **BR-RECUR-001**：每个周期规则、每个计划日期最多一个执行实例。
- **BR-RECUR-002**：自动模式在实例生成成功后创建普通账目。
- **BR-RECUR-003**：确认模式先创建待确认实例，用户确认后才创建账目。
- **BR-RECUR-004**：Worker 重试不得重复创建实例或账目。
- **BR-RECUR-005**：达到结束日期或总期数后自动标记完成。
- **BR-RECUR-006**：暂停规则不产生新的到期实例。
- **BR-RECUR-007**：恢复规则重新计算不早于当前日期的下一执行日。
- **BR-RECUR-008**：共同周期规则只能关联情侣账本，双方均可见。
- **BR-RECUR-009**：任务失败记录错误、重试次数和最后执行时间。
- **BR-RECUR-010**：暂停后恢复不补暂停期间账期；下一执行日重新计算为当天或之后符合规则的最近一期。
- **BR-RECUR-011**：月度和年度计划日始终以规则开始日期为锚计算；目标月份不存在锚点日时取月末，
  2 月 29 日年度规则在非闰年取 2 月末，并在后续闰年恢复到 2 月 29 日，不得按上期结果逐期漂移。
- **BR-RECUR-012**：执行实例首次成功物化时消耗一个计划期数并推进规则；确认模式的待确认实例已计入期数，
  后续确认或跳过不得再次增减期数。跳过只允许待确认实例，标记 `SKIPPED` 且不生成普通账目。
- **BR-RECUR-013**：确认金额只覆盖当前执行实例及其来源账目，不得修改规则默认金额或未来实例；相同用户、
  幂等键和确认金额重放原结果，不同金额复用幂等键必须冲突。
- **BR-RECUR-014**：个人和情侣有效成员均可创建所属账本的周期规则；双方可查看情侣规则，规则创建人可写，
  账本所有者可管理全部情侣规则和实例。退出成员、非成员和已解散账本不得继续访问或执行。
- **BR-RECUR-015**：周期规则创建按创建者和幂等键隔离；相同规范载荷重放原结果，不同载荷或已删除原结果
  返回幂等冲突。规则更新不得改写既有实例，总期数不得小于已物化期数；已有实例后不得修改收支类型、
  分类、频率、间隔、开始日期或生成方式。
- **BR-RECUR-016**：自动生成、确认入账和规则进度更新必须复用同一领域服务并在数据库事务中完成；生成账目
  的创建人固定为规则创建人，分类必须在执行时仍属于有效账本、类型匹配且启用。
- **BR-RECUR-017**：Worker 按规则创建人当前时区扫描到期计划，并以规则和计划日组成稳定任务键；执行时必须
  在领域锁内复核计划日。确认模式实例首次进入待确认状态时，向当前仍可管理该规则的活跃成员各创建一条通知，
  即规则创建人及情侣账本所有者，身份重合时去重；重试不得重复通知。

## 工资

- **BR-SALARY-001**：同一工资档案和工资月份只能有一条有效记录。
- **BR-SALARY-002**：应发工资等于所有收入项目之和。
- **BR-SALARY-003**：个人扣除等于所有个人扣除项目之和。
- **BR-SALARY-004**：实发工资等于应发工资减个人扣除。
- **BR-SALARY-005**：工资未到账时不得生成普通收入。
- **BR-SALARY-006**：到账生成的普通收入金额必须等于实发工资。
- **BR-SALARY-007**：重复确认到账不得重复生成收入。
- **BR-SALARY-008**：工资、社保、医保、公积金和个税仅本人可见。
- **BR-SALARY-009**：默认工资余额周期从到账日至下一预计发薪日前一天。
- **BR-SALARY-010**：日均可用金额仅在剩余天数大于零时计算。
- **BR-SALARY-011**：年度医保和公积金展示值来自工资记录，不代表官方账户余额。
- **BR-SALARY-012**：发薪日为 29 至 31 日而目标月份没有该日期时，预计发薪日使用该月最后一天。
- **BR-SALARY-013**：MVP 每位用户最多有一个未删除且状态为 `ACTIVE` 的工资档案；工资档案、默认项目模板、
  月度记录和项目只允许所属用户读取或维护，可猜测 ID 统一按资源不可见处理。
- **BR-SALARY-014**：工资档案默认项目模板与档案在同一事务内创建或替换；模板金额允许为零，模板变化不得
  回写既有月度记录。月度工资项目只保存正整数分，同一记录内 `itemCode` 唯一且名称、代码必须规范化。
- **BR-SALARY-015**：创建月度工资记录必须提交非空项目，或明确复制同档案紧邻上一个自然月的有效记录；
  两种来源互斥，复制不得跨过缺失月份查找更早记录。
- **BR-SALARY-016**：工资档案创建和月度记录创建使用“用户 + 操作 + 幂等键”及不可变规范载荷哈希；
  相同键同载荷重放原结果，不同载荷返回幂等冲突。不同键创建同档案有效月份仍返回月份重复冲突。
- **BR-SALARY-017**：工资记录汇总只由服务端从项目计算；`grossCent` 为 EARNING 合计，`deductionCent` 为
  DEDUCTION 合计，扣除不得大于应发，`netCent = grossCent - deductionCent`。项目替换与汇总更新必须同事务提交。
- **BR-SALARY-018**：已到账工资记录的月份、项目和汇总不可修改。工资余额中的固定支出定义为工资周期内本人
  个人账本中来源为 `RECURRING_RUN` 的有效支出，其余有效个人支出归日常支出；本口径不包含情侣账本支出。
- **BR-SALARY-019**：确认工资到账必须提交有效到账业务日期、是否同步收入和到账专用幂等键。相同用户、
  到账操作、幂等键和规范载荷重放原结果；同一键对应不同记录、日期或同步选项时返回幂等冲突。工资到账是
  一次性终态，不提供取消或改期；已由其他幂等键确认的记录返回已到账冲突。
- **BR-SALARY-020**：`syncEntry=false` 时到账记录不关联普通收入；`syncEntry=true` 时必须在本人有效个人账本
  的启用 `income.salary` 分类下生成且仅生成一条 `INCOME + SALARY` 来源明细，金额等于实发工资、业务日期
  等于到账日期、来源 ID 等于工资记录 ID。到账状态、来源明细、幂等事实和审计必须同事务提交或全部回滚；
  到账事实及同步收入的金额、日期、收支类型、归属和来源字段到账后不可修改，工资和来源明细仍仅本人可见。

## 攒钱目标

- **BR-SAVING-001**：目标金额必须大于零。
- **BR-SAVING-002**：目标已存金额等于初始金额加有效存入记录合计。
- **BR-SAVING-003**：存入记录新增、修改或删除与目标汇总更新必须在同一事务中完成。
- **BR-SAVING-004**：已存金额达到或超过目标金额时自动标记完成。
- **BR-SAVING-005**：情侣目标只能关联有效情侣账本。
- **BR-SAVING-006**：情侣贡献按存入记录的用户归属统计。
- **BR-SAVING-007**：完成状态不删除历史存入记录。
- **BR-SAVING-008**：目标初始金额归创建者；已完成目标的有效已存金额回落到目标金额以下时恢复为进行中。

## 通知、审计与导出

- **BR-NOTIFY-001**：通知必须归属于明确用户。
- **BR-NOTIFY-002**：用户只能读取和更新自己的通知。
- **BR-AUDIT-001**：删除、解散、权限变化、工资到账和任务手工重试必须写审计日志。
- **BR-EXPORT-001**：导出只能包含当前用户有权访问的数据。
