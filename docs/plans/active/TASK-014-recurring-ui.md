## 目标

- 实现 `/recurring`、`/recurring/new`、`/recurring/:id` 三条受保护移动端路由。
- 支持可见账本切换、周期规则列表、创建、详情、编辑、暂停、恢复和二次确认删除。
- 展示规则执行实例，并支持服务端允许的待确认实例按实际金额确认入账或二次确认跳过。
- 保持创建和确认失败重试使用稳定幂等键，快速重复点击只发送一次有效写请求。
- 在 320px、375px、480px 和日间/夜间下覆盖加载、空、错误、无权限、长文本、提交中和刷新恢复。

## 非目标

- 不修改后端 API、OpenAPI、数据库迁移、周期 Worker 或通知事实。
- 不实现站内通知读取/已读、人工任务重试或最小管理后台。
- 不把周期汇总加入首页或统计，不实现工资、攒钱、导出等后续模块。
- 未确认高保真前不新增品牌插画或独立视觉语言，只按既有低保真信息层级与设计 Token 实现。

## 关键设计

- 周期 API 客户端基于共享 OpenAPI 类型，规则和实例列表按 100 条分页顺序完整读取并设置最大 100 页边界。
- 列表账本选择写入 `ledgerId` 查询参数并持久化；实例通过响应中的 `rule.ledgerId` 归入当前账本。
- 列表先显示待确认实例，再显示规则；不使用部分数据计算伪造月度固定支出汇总。
- 创建表单覆盖账本、收支、金额、分类、频率/间隔、开始日期、自动/确认、结束日期或总期数、提前提醒天数。
- 详情编辑复用同一表单；后端 `RECURRING_SCHEDULE_LOCKED`、分类失效和状态冲突映射为可操作提示并刷新事实。
- 创建幂等键在成功后才轮换；每个待确认实例保持独立确认幂等键，失败不轮换，成功后移除。
- 暂停、恢复、删除、确认和跳过期间禁用同一操作；删除和跳过使用二次确认。
- 返回目标只接受 Router 解析后属于周期列表的站内 URL，避免开放重定向。

## 风险

- 规则或实例超过单页时漏加载，会隐藏待确认事项或错误展示空状态。
- 确认失败后更换幂等键，可能把未知成功的网络请求再次入账。
- 前端按当前用户或账本角色自行计算能力，会扩大情侣成员权限。
- 已有实例的规则核心计划字段被编辑时会被服务端拒绝，客户端必须保留表单并解释冲突。
- 原生日期/选择控件、抽屉和长规则名在 320px 或暗色主题可能溢出或出现白底。
