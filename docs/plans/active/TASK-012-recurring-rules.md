## 关键设计

- 月/年计划日始终以规则 `startDate` 为锚计算；短月取月末，闰日按目标年份月末处理，避免逐期漂移。
- 每次首次物化计划实例即消耗一个计划期数；确认模式的 `PENDING`、自动模式的 `GENERATED` 以及后续
  `SKIPPED` 都不重复增减规则期数。
- 跳过只允许 `PENDING` 确认实例，标记 `SKIPPED` 且不生成账目；不改变规则模板或后续日期。
- 确认金额只覆盖当前实例并作为入账金额，不回写规则默认金额；确认按用户、操作和幂等键校验重放。
- 个人/情侣有效成员均可创建和查看规则；创建人可写自己的规则，账本 OWNER 可管理全部规则和实例。
- 规则创建使用不可变请求哈希；实例依靠 `(ruleId, scheduledDate)`，来源账目依靠
  `(sourceType, sourceId)`，确认依靠用户/幂等键共同保证并发安全。
- 规则更新不改写既有实例；已有实例后锁定收支分类、频率、开始日期和生成方式。活动规则从用户业务日期
  当天或之后重新计算下一计划日，并且总期数不得小于已物化期数。

## 风险

- 以“上一次执行日”递推会让 31 日规则永久漂移到 28 日，必须从开始日锚点计算。
- 规则进度、实例和来源账目若分事务提交，会出现重复入账或期数丢失。
- 情侣成员退出、账本解散、创建人禁用后必须在查询和执行时重新校验，不能只信历史外键。
- 已生成实例引用停用分类仍需可读，但新实例入账必须明确失败，不能改用其他分类静默记账。
- 删除规则不能删除或解绑既有来源账目。
