# TASK-016：工资到账与收入联动

## 状态

已完成并正式关闭

## 模型与推理

- 推荐模型：Sol
- 推理：High
- 是否使用子代理：否

## 目标

- 实现本人私有工资记录的到账确认，显式接收到款日期、是否同步普通收入和请求幂等键。
- 同步收入时，在用户有效个人账本中生成唯一 `SALARY` 来源收入明细，金额等于实发工资，业务日期等于到账日期。
- 工资到账状态、来源明细、幂等事实和审计记录在同一数据库事务内提交或回滚。
- 固化重复请求、并发确认、跨用户访问和到账后不可修改的数据边界。

## 非目标

- 不实现工资余额、固定/日常支出和年度统计；属于 TASK-017。
- 不实现工资手机页面；属于 TASK-018。
- 不提供取消到账、改到账日期、补挂或解绑收入明细的接口。
- 不修改现有工资档案、月度记录创建/编辑、Entry 手工录入、借贷或周期任务行为。

## 决策记录

- 到账是一次性终态操作，不提供反向取消；同一到账幂等键和规范载荷重放返回既有结果，键复用到不同载荷时冲突。
- `syncEntry=false` 只确认到账并保留 `entryId=null`；`syncEntry=true` 必须在有效个人账本和启用的 `income.salary` 分类中原子创建收入明细。
- 来源收入固定为 `INCOME + SALARY`，金额等于 `netCent`，业务日期等于 `paidDate`，创建者和个人账本所有者均为工资记录本人。
- 工资记录与同步收入采用双向数据库约束；到账事实和 `SALARY` 来源明细的财务/来源字段到账后不可变。
- 对迁移前已经到账的数据回填保留型幂等事实；这类历史键不作为客户端重放契约。

## 风险

- 并发不同幂等键确认同一记录可能生成重复收入，需要记录级事务锁、来源唯一约束和终态检查共同保护。
- 只校验单边外键可能出现工资显示已关联但 Entry 来源不匹配，必须建立双向延迟约束。
- 同步分类或个人账本不可用时若先写状态会留下半完成数据，所有写入必须在同一事务内。
- 已到账记录或来源明细若允许后改金额/日期，会破坏财务一致性，数据库触发器必须阻断绕过 API 的修改。
- 错误响应必须保持防枚举，跨用户记录统一返回 404。
